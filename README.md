# deep-verify

> Claude(Anthropic) + Codex(OpenAI)를 같이 쓰면서, **"한 모델만 보면 놓치는 걸 다른 회사 모델·다른 관점은 잡더라"**는 경험에서 만든 Claude Code 검증 스킬.

코드 변경을 머지하기 직전에, **세 방향(실제 테스트 실행 + 다른 회사 모델 교차검증 + 그 repo의 불변식 감사)**으로 동시에 검증하고, 정해진 규칙으로 합쳐 **"확실히 확인된 것 / 의견뿐인 것"을 절대 섞지 않고** 보고합니다.

> 세 번째 방향(레인 C)은 원래 "다관점 적대 리뷰(고정 5렌즈)"였는데, 실사용 기록을 세어보니 **코드 리뷰에서 단독 적발이 0건**이었습니다(같은 회사 모델이라 저와 사각지대를 공유합니다). 반대로 **그 프로젝트가 명문화한 규칙을 어겼는지 감사하는 일**은 외부 모델이 원리적으로 못 하는 것이라, 거기로 역할을 바꿨습니다. 자세한 근거는 `SKILL.md`의 실측 각주와 2026-07-09 갱신 이력에 있습니다.

**Status:** Claude Code 스킬 문서·절차 (실행 러너/패키지 아님) · `Last reviewed: 2026-07-17` against default-branch source.

## 왜 만들었나

저는 Claude랑 Codex를 같이 쓰는데 — 한쪽(Claude **또는** Codex)에만 매몰되면, **같은 모델이 같은 걸 또 봐도 관점이 같아서** 그 모델이 못 잡는 버그·에러는 계속 못 잡더라고요. 그런데 **다른 회사 모델이, 다른 관점에서 보면** 보이는 버그·에러가 있었어요.

원래는 Claude ↔ Codex 교차검증 워크플로우(`cross_verify`, **not publicly available** — 공개 링크·내부 구현 설명 없음)에서 한 발 더 나가, **실행 증거 + 다른 회사 모델 교차검증 + repo 불변식 감사**를 한 번에 합치는 스킬을 만든 게 이 `deep-verify`입니다.

개인적으로 편하게 쓰고 있어서 공개합니다.

## 어떻게 작동하나

변경분/구현을 **세 레인으로 동시에** 검증하고, **고정된 결정 규칙**으로 합칩니다:

1. **결정적·실행 (레인 A)** — 테스트·빌드를 *말로만 말고 실제로 돌려서* 통과/실패를 **사실**로 캡처
2. **교차검증 (레인 B)** — **다른 회사 모델(Codex 등)**에게 read-only 코드 리뷰 (같은 회사 모델은 나와 같은 사각지대를 공유하니 의미가 적음). 가능하면 2사 블라인드 병렬; 합의 자체는 검증이 아니라 "신호 없음"으로 취급
3. **repo 불변식 감사 (레인 C)** — 고정 5렌즈 재사용이 **아님**. 대상 repo 규범 문서(`CLAUDE.md`·설계문서·CONTRIBUTING 등)에 **명시된 절대규칙**을 렌즈로 뽑고(규칙 원문 인용 못하면 렌즈를 돌리지 않음), 여기에 고정 렌즈 2개(**실효**, **테스트 teeth**)를 더해 **3~5개**만 돌림. 이번 diff에 안 닿는 렌즈는 skip. 대상이 설계 문서이거나 명문화된 불변식이 없을 때만 일반 5렌즈(정확성/보안/재현성/계약·인터페이스/책임·경계)로 폴백. 각 렌즈는 confirm이 아니라 **refute**를 시도

→ 그리고 제일 중요한 규칙: **"체크할 게 없으면 절대 초록불(통과)로 안 칠한다."** **초록불 ≠ 참.** 결정적으로 확인된 것과 의견뿐인 것을 두 칸으로 나눠 보고합니다.

레인들이 **각자 따로 보고만 하면 같은 사각지대를 공유**해서 놓치는 게 생기더라고요. 그래서 관점이 **엇갈리거나** 보안·데이터 같은 **치명 지적**이 나오면, 서로 근거(코드 위치·테스트·명세)를 대고 **짧게 반박을 주고받은 뒤** 정리합니다 — 말 잘하는 쪽이 아니라 근거 있는 쪽이 남게요. 그래도 안 갈리면 스킬이 억지로 결론 내지 않고 **제가 판단하도록** 남겨둡니다. 중요도도 "이 검증이 뭘 위한 건지(stake)"에 따라 갈리는 건 제 몫으로 넘겨요.

가중치: **결정적(레인 A) > 교차(레인 B) > 같은-회사 렌즈(레인 C)**. 단 레인 C가 낸 **그 repo 명문화 불변식 위반**은 외부 모델이 원리적으로 못 아는 지적이므로 일반 정확성처럼 깎지 않습니다.

## 언제 쓰면 좋나

모든 작업에 의심을 갖는 건 아니에요. 다만 **"뭔가 확인해야 직성이 풀리는 성격"**이라면 — 공들인 코드를 머지/배포하기 직전, 틀리면 곤란한 데 도움이 될 거예요. (오타·한 줄 같은 작은 수정엔 과합니다.)

## 한계 · 네트워크

- 이 repo 자체에는 테스트 러너·CI 설정이 없습니다. 레인 A는 **검증 대상 프로젝트** 쪽에서 명령을 실행합니다.
